# Garden App — Kiro Spec Document

## Overview

A mobile-first Flutter app that helps gardeners manage their plants, track care history, follow AI-generated watering schedules, and diagnose plant health issues using photos. The app is backed by an AI agent that integrates live weather data, per-plant history, and user-submitted care instructions to generate personalised, context-aware advice.

---

## Tech Stack

| Layer | Technology |
|---|---|
| Frontend | Flutter (mobile-first, iOS + Android) |
| Database | AWS DynamoDB |
| File storage | AWS S3 |
| AI agent | AWS Lambda + Claude (Anthropic) |
| Weather | External weather API (integrated by agent) |
| Auth | AWS Cognito |

## Data Model

### Core Tables (DynamoDB)

**USER**
- `user_id` (PK, UUID)
- `name`, `email`, `location`, `created_at`

**PLANT_SPECIES**
- `species_id` (PK, string — e.g. `"prunus-domestica"`)
- `common_name`, `latin_name`, `family`, `climate_zone`, `description`

**PLANT_INSTANCE**
- `instance_id` (PK, UUID)
- `user_id` (FK), `species_id` (FK)
- `nickname`, `location_in_garden`, `planted_at`, `status`, `updated_at`

**WATERING_SCHEDULE**
- `schedule_id` (PK, UUID)
- `instance_id` (FK)
- `base_interval_days` — species default watering frequency
- `rain_skip_threshold_mm` — skip watering if rain exceeds this
- `hot_day_threshold_celsius` — temperature above which interval shortens
- `hot_day_adjustment_days` — days to subtract from interval on hot days
- `rain_adjustment_days` — days to add after rain above threshold
- `soil_type` — affects drainage calculation
- `next_water_date` — computed by agent after each watering or weather check
- `last_watered_at`

**CARE_INSTRUCTION**
- `instruction_id` (PK, UUID)
- `species_id` (FK), `submitted_by_user` (FK)
- `title`, `body`
- `source_type` — enum: `RHS`, `Forum`, `Book`, `Other`
- `source_url`
- `scope` — enum: `species`, `instance`, `garden`
- `ai_verified` (boolean), `ai_confidence` (float)
- `created_at`

**ACTION_LOG**
- `action_id` (PK, UUID)
- `instance_id` (FK), `triggered_by_instruction` (FK, nullable)
- `action_type` — enum: `watered`, `pruned`, `fertilised`, `issue_found`, `repotted`, `other`
- `notes` (free text)
- `severity` — nullable, enum: `mild`, `moderate`, `severe` (used for `issue_found`)
- `logged_by` — enum: `user`, `agent`
- `image_refs` — array of S3 keys
- `weather_snapshot` (JSON) — weather at time of action
- `ai_context_snapshot` (JSON) — agent reasoning at time of action
- `occurred_at`

### Image Storage (S3)

Key pattern: `users/{user_id}/plants/{instance_id}/{timestamp}_{action_id}.jpg`

Store original full-resolution images. Apply client-side compression before upload. Attach lifecycle policy to archive to S3 Glacier after 6 months.

## Navigation Structure

Bottom tab bar with four tabs:

1. **Home** — Dashboard with AI garden summary and plant grid
2. **Water** — Watering schedule and today's tasks
3. **Plants** — Browse all plant instances (leads to plant detail)
4. **Profile** — User settings

## Screens

### 1. Home — Dashboard

**Purpose:** Daily overview. The first thing the user sees each morning.

**Layout (top to bottom):**

- Dark green header with greeting (`Good morning, {name}`) and date
- **AI summary card** — generated each session by the agent. Pulls together: weather forecast, pending watering, any active issues, care actions due. Displayed with an "AI summary" badge. Shows 2–3 weather chips (today's temperature, tomorrow's forecast).
- **Alert card** (conditional) — shown only when there is an active unresolved issue on any plant. Amber background. Shows plant name, issue type, days since logged, and one-line AI recommendation. Persists until the issue is marked resolved.
- **Plants section header** — shows total plant count, "+ Add" link
- **Plant grid** — 2-column grid of plant cards. Each card shows: emoji/icon, nickname, species name, status dot + label. Status dot colours: green (thriving), amber (needs attention), red (alert). Tapping a card navigates to Plant Detail.

**Data required:**
- Latest AI summary (generated by agent on session open)
- All `PLANT_INSTANCE` records for the user
- Active unresolved `ACTION_LOG` records of type `issue_found`
- Current weather snapshot

### 2. Plant Detail

**Purpose:** Full information view for a single plant instance.

**Header (dark green):**
- Back button to garden
- Plant emoji, nickname, planted date, garden location
- Status chips: current status, next watering, sun requirements

**Tabs (below header):**
- Overview
- History
- Photos
- Care

#### 2a. Overview Tab

- **AI personalised insight card** — green tinted. Pulls from multi-season history for this instance. References specific past events (e.g. heat stress last July, previous mildew recovery). Generated by agent per-instance.
- **Care history timeline** — chronological list of `ACTION_LOG` entries. Each entry shows: date, action type, notes, tag chip (colour-coded by type), and whether it was user-logged or AI-suggested. Entries are distinct between user and agent actions.
- **Care instructions section** — list of `CARE_INSTRUCTION` records scoped to this species or instance. Each shows: title, body, source, AI verification badge. "+ Add yours" link opens Add Care Instruction sheet.

#### 2b. History Tab

Full paginated timeline of all `ACTION_LOG` entries for this instance.

#### 2c. Photos Tab

See Screen 5 (Photo Gallery).

#### 2d. Care Tab

Full list of care instructions for this species. Includes AI-verified and user-submitted instructions. Filterable by source type.

**Actions:**
- "+ Log action" button (floating or in header) opens Log Action sheet (Screen 4)

---

### 3. Watering Schedule

**Purpose:** Dedicated tab for managing all watering tasks across the garden.

#### 3a. Today view (default)

**Header:**
- Date, title "Watering"
- Rain chip — shown when agent has auto-adjusted schedules due to weather

**AI skip card (conditional):**
- Shown when agent has skipped one or more plants due to recent rainfall
- Lists affected plants and reason (e.g. "12mm rain yesterday")

**To water today list:**
- Each row: circle tap target (mark as done), plant name, sub-label (days overdue or "Due today", base interval), badge (Overdue / Due / Skipped)
- Tapping the circle check marks the plant as watered. This creates an `ACTION_LOG` entry with `action_type: watered`, `logged_by: user`, and a `weather_snapshot`. The agent then recalculates `next_water_date` for that instance.
- "Skip all" link skips all pending for today with a single tap.

**Coming up list:**
- Next 5 upcoming watering dates across all plants
- Each row: dot (coloured by plant), plant name, date, weather adjustment label if applicable (e.g. "+2d rain")

#### 3b. Per-plant calendar view

- Horizontal scrollable chip row to switch between plant instances
- **Schedule config card** — shows: base interval, weather adjustment rules, skip threshold, soil type. Read-only; editable via settings.
- **Monthly calendar grid** — days colour-coded: watered (dark green), scheduled upcoming (light green), skipped by AI (light blue), today (dark green filled). Tapping a past day shows the action log entry for that date.
- **Legend** below calendar
- **Watering history log** — last 5 watering entries with date and volume (if logged)

### 4. Log Action Sheet (Bottom Sheet)

**Purpose:** Logging a care action against a specific plant instance. Triggered from Plant Detail header or from the watering schedule "mark done" flow.

**Sheet header:**
- Plant icon, plant nickname, garden location, today's date
- Close button
- When opened from the watering schedule for a "mark watered" action, the sheet pre-selects "Watered" and the AI conversation is skipped unless the user adds a note.

**Action type grid (2 columns, 6 options):**

| Action | Icon colour | Notes placeholder |
|---|---|---|
| Watered | Blue | "e.g. watered 2L at base, soil was quite dry..." |
| Pruned | Green | "e.g. removed 4 lateral shoots, plant looking vigorous..." |
| Fertilised | Amber | "e.g. half-strength tomato feed, first flowers forming..." |
| Issue found | Red | "e.g. white powdery coating on lower leaves, two affected..." |
| Repotted | Green | "e.g. moved to 30cm pot, used peat-free compost..." |
| Other | Neutral | "Describe what you did..." |

- Selecting an action type highlights the card and updates the submit button label and notes placeholder.
- Selecting "Issue found" reveals a **Severity picker**: Mild / Moderate / Severe (colour-coded chips).

**Detail section:**
- Notes field (optional, free text)
- Photo attachment (optional) — camera icon opens camera; attached photos shown as thumbnails

**Submit row:**
- "Draft" button — saves as draft, not confirmed to action log
- "Log {action type}" primary button — confirms and writes to `ACTION_LOG`

#### 4a. AI Follow-up Conversation

After submitting any action, the sheet transitions from the form into a **lightweight chat thread** within the same sheet. The sheet header switches from "form mode" to "logged mode" (shows green "Issue logged" badge).

The AI agent responds in the thread with a contextual follow-up:
- For `issue_found`: asks clarifying questions (location on plant, visual description) to help diagnose
- For `pruned` / `fertilised`: may acknowledge and note impact on next care schedule
- For `watered`: brief acknowledgement, updates next watering date silently

**Chat thread UI:**
- AI messages: left-aligned bubble with AI avatar, "Garden AI" sender label
- User messages: right-aligned dark green bubbles
- AI can ask for a photo within the conversation. Camera button is always visible in the reply input row.
- User can attach a photo as a reply — shown as an image bubble with optional caption
- AI analyses the photo (vision call) and responds with a structured **diagnosis card** when relevant. Diagnosis card includes: condition name, confidence, description, progression note if prior photos exist. Two action buttons: "Add treatment to schedule" and "View care instructions".
- Thread is limited to the immediate conversation about this action. It is not an open-ended general chat.
- All photos sent in the conversation are saved to S3 and their keys are appended to the `ACTION_LOG` record's `image_refs` field.

### 5. Photo Gallery (Plant Detail — Photos Tab)

**Purpose:** Visual timeline of all photos for a specific plant instance.

**Filter row:**
- All / Issues / Healthy / After care — filters by action tag

**Photo grid:**
- 3-column grid, grouped by month with month label headers
- Each thumbnail shows: action tag chip (Issue / Healthy / Pruned etc.), AI analysis indicator dot (shown when AI has analysed this photo)
- Tapping a photo opens Photo Detail view
- "+ Add photo" cell always shown as the last item in the grid

#### 5a. Photo Detail View

**Full-width photo display:**
- Overlaid gradient at top and bottom
- Top bar: back button, add note button
- Bottom left: date and action context (e.g. "Issue logged · Moderate")
- Bottom right: photo counter (e.g. "1 / 6")

**Thumbnail strip:**
- Horizontal scrollable strip of all photos for this plant, oldest to newest
- Each thumbnail has a coloured dot matching its action type
- Active photo is highlighted with a green border

**AI analysis card (conditional):**
- Shown when the AI has analysed this photo
- For issues: diagnosis name, confidence, written analysis
- **Progression comparison**: if a prior photo exists for the same condition, shows a side-by-side two-thumbnail comparison with dates
- Progression is described in plain language (e.g. "Coverage has increased roughly 30% since 5 Apr")

**Action pills:**
- "Log treatment done" — opens Log Action sheet pre-filled
- "View care plan" — navigates to Care tab for this plant

### 6. Add Care Instruction Sheet (Bottom Sheet)

**Purpose:** User submits a care instruction they found externally. Triggered from "+ Add yours" on the Care tab.

**AI review notice** at top of sheet — informs user the instruction will be reviewed by the AI agent and cross-referenced with plant history before being added.

**Fields:**
- Instruction title (text input)
- Details (multiline text input — when and how to apply)
- Source type (chip selector): RHS / Forum / Book / Other
- Source URL (optional text input)
- Applies to (chip selector): this specific plant / all of this species / entire garden

**Submit button:** "Submit for AI review"

**Agent behaviour after submission:**
- Agent reviews instruction against known species data
- Sets `ai_verified` and `ai_confidence` on the record
- Low-confidence instructions are stored but flagged; agent adds a caveat when referencing them in care summaries
- User is notified when the review is complete

## AI Agent Behaviour

### Session open
On app open, the agent:
1. Fetches latest weather data for the user's location
2. Recalculates `next_water_date` for all plant instances based on weather
3. Checks all `ACTION_LOG` entries for unresolved issues
4. Generates the home screen AI summary

### Watering logic
- Base interval set per plant in `WATERING_SCHEDULE`
- If forecast rain > `rain_skip_threshold_mm`: push `next_water_date` forward by `rain_adjustment_days`
- If today's temperature > `hot_day_threshold_celsius`: reduce interval by `hot_day_adjustment_days`
- Agent logs auto-skips as `ACTION_LOG` entries with `logged_by: agent`

### Personalised insight generation
- Per plant instance, agent pulls full `ACTION_LOG` history + any prior season data
- Identifies patterns: recurring issues, seasonal underperformance, previous interventions and their outcomes
- Generates 2–3 sentence insight shown on Overview tab

### Photo analysis
- Triggered when user attaches a photo to a log action or chat reply
- Agent calls vision model with the image + plant species context + recent action log
- If prior photos exist for the same plant, agent retrieves them from S3 for comparison
- Returns: condition name, confidence, written analysis, progression delta if applicable

### Care instruction review
- Triggered on submission of a new `CARE_INSTRUCTION`
- Agent checks instruction against species data and user's instance history
- Sets `ai_verified: true/false` and `ai_confidence` float
- Adds a note to the instruction body if confidence is below threshold

## Key UX Principles

- **Watering mark-as-done is one tap** — the circle check on the today list is the primary interaction. No confirmation dialog.
- **AI follow-up is optional** — after logging an action the user can dismiss the sheet without engaging the chat thread.
- **Photos are never mandatory** — the camera is always available but never required to complete a log entry.
- **AI and user actions are visually distinct** — in all timeline and history views, agent-logged entries are clearly labelled differently from user-logged entries.
- **Issue alerts persist** — the alert card on the home screen stays visible until the issue is explicitly resolved, not just acknowledged.
- **Draft support** — any log action can be saved as a draft and confirmed later, useful for quick garden observations.
